# 前端开发工作流

## 目标

在明确范围下完成前端改动，并提供可预期的质量校验与清晰交付说明。

## 触发场景

当需求涉及以下内容时使用本 skill：

- React 组件或页面开发
- TypeScript 前端逻辑改动
- UI 缺陷修复
- 样式或布局调整
- 前端重构
- 基于 Vite 的前端任务

## 执行流程

按以下顺序执行。

### 1) 明确任务与约束

- 用一句话复述目标结果，确认理解一致。
- 识别关键约束（框架、样式规范、既有模式、API 契约）。
- 若需求不完整，先提出聚焦问题再开始改动。

### 2) 先检查相关代码

- 定位已存在的相关文件与实现模式。
- 优先保持命名、结构、样式方式与项目一致。
- 引入新抽象前，先复用现有工具函数、hooks、组件。

### 3) 规划最小可行改动

- 选择最小且安全的改动满足需求。
- 仅修改必要文件，控制影响面。
- 避免无关清理，除非会阻碍交付。

### 4) 实施改动

- 保持代码可读性与类型明确。
- 仅在不直观逻辑处添加简短注释。
- 除非需求明确要求破坏性变更，否则保持向后兼容。

### 5) 本地验证

- 尽量执行与改动范围匹配的校验（lint/test/build）。
- 交付前修复本次改动引入的问题。
- 若无法完成全量验证，明确说明未执行项及原因。

### 6) 反馈结果

- 说明改了什么以及为什么改。
- 列出受影响文件。
- 给出验证状态与后续建议（如有）。

## 输出格式

完成实现后使用以下结构：

```markdown
已完成 [feature/fix/refactor]，范围：[scope]。

改动内容：
- [关键改动 1]
- [关键改动 2]

涉及文件：
- `path/to/file1`
- `path/to/file2`

验证结果：
- [x] [已通过的命令/检查]
- [ ] [未执行的检查 + 原因]

备注：
- [风险、假设或后续建议]
```

## 前端质量检查清单

交付前确认：

- 类型正确（避免可避免的 `any`）。
- UI 状态完整（按需处理 loading/empty/error）。
- 覆盖输入与异步流程的边界情况。
- 遵循既有设计与样式规范。
- 不引入死代码与无关改动。

## 默认偏好

- 优先组合式设计，避免超大单体组件。
- 对公开 props 与共享工具优先使用显式类型。
- 优先针对任务的小范围重构，避免大规模重写。
